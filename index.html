What is SPDY?

  • An experimental protocol from Google
  • The starting point for IETF's HTTP 2.0 standardisation work
  • Not an acronym
  • Pronounced 'speedy'
  • No information on Safari

Who supports it?

Browsers:
Chrome (desktop, iOS, Andriod), Firefox, Opera, IE 11+ (!)

Websites:
Google, Facebook, Twitter, GitHub, etc.

Application servers/platforms:
Apache (mod_spdy), Nginx, Ruby, Python, NodeJS, Java (Jetty and Netty), etc.
  • Browsers open up to 6 connections (was 2)
  • These connections take time to warm up
  • TCP slow start important: Initial congestion window (init_cwnd)
  • Redundant User-Agent, etc.

What is wrong with HTTP 1.1?

Not designed to minimise latency:

  • Single request per connection
  • Only client-initiated requests
  • Headers are not compressed
  • Optional data compression
  • Redundant headers
SPDY's purpose is to improve end-user perceived latency.
  • It's binary!
  • Not much support for request prioritisation.

How does SPDY achieve that?

  • Multiplexing requests on a single connection
  • Header compression
  • Server push
  • Server hint
  • Request prioritisation

  • CRIME attack: Compression and encryption don't go well together.

Server hint

More polite than push

Request:

GET /index.html HTTP/1.1
Host: www.server.com

Response:

HTTP/1.1 200 OK
Content-Type: text/html
Link: </jquery.js>; rel=prefetch
